\subsection{Semantics :: of the Reagent Language}
\label{subsec:semantics}

The semantics of the Reagent language defines all possible executions of a concurrent program consisting of multiple threads interacting through a shared memory. Formally, the semantics are represented by a labeled transition system $\mathcal{S} = (\Phi,\rightarrow)$, where $\Phi$ is the set of program configurations and \(\rightarrow\) is the
transition relation generated by the rules shown in
Figure~\ref{fig:semantics}.
Each configuration $C$ in $\Phi$ represents the global state of the program at a particular point during execution. Formally, a configuration is a pentuple $C = (\sigma,\lambda,E,\hat{\sigma},\hat{E})$, where:

\begin{itemize}

\item \(\sigma : \Loc \rightarrow \Val\) maps each shared memory location to its current value.

\item \(\lambda \in \Tid \cup \{\bot\}\) records the \emph{token} of the unique transaction that is currently live. $\lambda = t$ means that the transaction opened by thread $t$ (whose token is $t$) is running, and $\lambda = \bot$ means that no transaction is active. Because at most one transaction runs at any time, $\lambda$ is a single global value, not a per-thread map: the shared state should not be indexed by thread identity.

\item $E : \Tid \rightharpoonup \Expr$ maps each active thread to the
expression currently being evaluated by that thread.

\item $\hat{\sigma}: \Loc \rightarrow \Val$ is the snapshot of the shared store taken when the live transaction began; it is defined only while $\lambda \neq \bot$, and is used to support transaction restart. Since at most one transaction is active, a single snapshot suffices.

\item $\hat{E} \in \Expr$ is the expression from which execution should resume when a retry operation is encountered; like $\hat{\sigma}$ it is defined only while $\lambda \neq \bot$, and a single saved expression suffices because only one transaction is ever live.

\end{itemize}

\caption{Semantics of Reagent commands}
\label{fig:semantics}
\end{figure}

Figure~\ref{fig:semantics} gives a formal description of the transition
rules. $\textrm{\sc{Get}}$ reads the value stored at a shared location
$l$. If thread $t$ holds the live transaction ($\lambda = t$) and $\sigma(l)=v$, then $\getc{l}$ returns the value $v$; the emitted action $\getev{l}{v}{t}$ records both the value $v$ that was read and the token $t$ of the transaction that read it. The store and transaction state remain unchanged.

$\textrm{\sc{Set}}$ updates a shared location. When thread $t$ (with $\lambda = t$) executes
$\setc{l}{v}$, the value stored at location $l$ is updated to $v$ and the
command returns the unit value $\unit$; the emitted action $\setev{l}{v}{t}$ records the written value $v$ and the token $t$.

$\textrm{\sc{Begin}}$ starts a transaction. No transaction may currently be live ($\lambda = \bot$); transactions are therefore mutually exclusive, with at most one live transaction across all threads at any time. $\textrm{\sc{Begin}}$ creates a new token $t$ for the transaction and records it by setting $\lambda := t$, and the current store and expression are saved in $\hat{\sigma}$ and $\hat{E}$ respectively. These saved values can later be used by $\textrm{\sc{Retry}}$.

$\textrm{\sc{End}}$ commits the current transaction, resetting $\lambda$ to $\bot$. The store is unchanged.


$\textrm{\sc{Retry}}$ is retrying with \emph{lock-releasing}. Instead of keeping the thread inside the transaction and spinning, it (i) rolls the store back to the snapshot $\hat{\sigma}$, so the abandoned attempt has no effect; (ii) sets $\lambda=\bot$, \emph{releasing the single critical section} so that another thread may now $\bbegin$ and run; and (iii) re-arms $t$'s expression as $\bbegin\,;\,\hat{E}$, so that when $t$ is next scheduled it must re-acquire the lock through $\textrm{\sc{Begin}}$ before re-running the saved body. Because $t$ relinquishes the lock and takes a fresh snapshot on its next $\bbegin$, a write committed by another thread in the meantime is visible to the new attempt.

\section{A memory model}
\label{sec:scheduling}

This question came up during a discussion with \textsc{Vimala}, wherein she
asked me about the memory model of \texttt{Reagents}. The following paragraph
is an attempt to capture the core idea behind it. Because I want
to ensure I have a decent understanding of the memory model  and weed out any misconceptions at a
nascent stage, I am writing it down here so that you can point out any mistakes
or gaps in my logic.


So far the rules describe \emph{which} steps are possible; they say nothing about
\emph{which thread} takes the next step. That choice belongs to a
\emph{scheduler}, which at each point picks one thread from $dom(E)$ and lets it
take a single transition. The transition relation is non-deterministic precisely
because the scheduler is free to choose; an execution of the program is one
particular sequence of scheduler choices.

Now suppose every thread runs its own transaction, i.e.\ each $E(t)$ has the
shape $\bbegin\,;\,\textit{body}_t\,;\,\eend$. Consider two threads $t_1$ and
$t_2$ and an interleaving:

\begin{itemize}
\item The scheduler picks $t_2$. Thread $t_2$ fires $\textsc{Begin}$: since
$\lambda = \bot$, the rule applies and sets $\lambda := t_2$. Thread $t_2$ has now
\emph{acquired the lock} --- it owns the single live transaction.
\item The scheduler preempts $t_2$ mid-transaction and selects $t_1$. Thread
$t_1$ is poised at its own $\bbegin$, but its $\textsc{Begin}$ rule requires
$\lambda = \bot$, whereas $\lambda = t_2$. The premise fails, so $t_1$ \emph{cannot
make progress}: there is only one lock, and it forbids $t_1$ from entering the
critical section while $t_2$ holds it. (Every tx rule
$\textsc{Get}$, $\textsc{Set}$, $\textsc{End}$, $\textsc{Retry}$  carries the
premise $\lambda = t$, so the only thread that can step transactionally is the
current lock holder $t_2$.)
\item Say a context switch happens, and the thread holding the
transaction, $t_2$, is brought back up. It continues until it either commits
($\textsc{End}$, which sets $\lambda := \bot$) or retries
($\textsc{Retry}$, which also releases the lock). Either way $\lambda$ returns to
$\bot$, and only then can $t_1$'s $\textsc{Begin}$ finally fire.
\end{itemize}

We assume the scheduler is \emph{fair}.

\subsection{Evaluation contexts}

An evaluation context $\ectx$ is an expression containing a single
hole $[\cdot]$ that marks exactly where the next reduction step occurs. To
avoid confusion, the notation $\ectx$ is used here in the
same spirit as the small-step evaluation relation $X \downarrow Y$ (signifying
that $X$ evaluates to $Y$).
I just wated to separate this one specific notation of evaluation context from $E$, which represents a mapping from
($\mathit{Tid}$) to expressions ($\mathit{Expr}$).
\[
\ectx \;::=\; [\cdot]
\;\mid\; \ectx\,;\,e
\;\mid\; \textsf{let}\;x = \ectx\;\textsf{in}\;e
\;\mid\; \ectx + e \;\mid\; v + \ectx
\;\mid\; \ectx == e \;\mid\; v == \ectx
\;\mid\; \textsf{if}\;\ectx\;\textsf{then}\;e\;\textsf{else}\;e
\;\mid\; \setc{l}{\ectx}.
\]
We write $\ehole{e}$ for the expression obtained by plugging $e$ into the hole. Thus
$E(t)=\ehole{\bbegin}$ asserts that the redex thread $t$ is about to execute is
$\bbegin$, sitting inside the surrounding continuation $\ectx$.

\subsection{The command rules in context form}

Every command rule now fires on its redex inside an arbitrary context $\ectx$,
preserving the continuation. The configuration is the pentuple
$\langle\sigma,\lambda,E,\hat{\sigma},\hat{E}\rangle$ of
Section~\ref{subsec:semantics}.

\begin{itemize}
\item $E(t)=\ehole{\bbegin}$ and $E'=E[t\mapsto \ehole{\unit}]$: the $\bbegin$ reduces to
$\unit$ \emph{in place}, leaving the continuation $\ectx$.
\item $\hat{E}'=\ehole{\unit}$: the saved resume point is $\ehole{\unit}$,
the body \emph{after} $\bbegin$, which is precisely what $\textsc{Retry}$
reinstates.
\end{itemize}
\textsc{Retry}rolls the store back to
the snapshot $\hat{\sigma}$, releases the lock ($\lambda = \bot$), and
re-arms $t$ as $\bbegin\,;\,\hat{E}$ so that the thread must re-acquire the
lock through $\textsc{Begin}$ before re-running. I tried to capture the idea of taking a fresh snapshot in the
process. Since, $\hat{E} = E_{\downarrow,0}\llbracket\unit\rrbracket$ is the body saved by
$\textsc{Begin}$ (where $E_{\downarrow,0}$ is the context at the opening $\bbegin$), the
re-armed expression $\bbegin\,;\,\hat{E}$ reconstructs the original
transaction and reduces back to $E_{\downarrow,0}\llbracket\bbegin\rrbracket$.

\subsection{Administrative reductions}


\[
\begin{aligned}
\ehole{\textsf{let}\;x = v\;\textsf{in}\;e} &\;\reduces\; \ehole{\subst{e}{x}{v}} &
\ehole{\unit \,;\, e} &\;\reduces\; \ehole{e} \\
\ehole{n_1 + n_2} &\;\reduces\; \ehole{n} \quad (n = n_1 + n_2) &
\ehole{\textsf{if}\;\textsf{true}\;\textsf{then}\;e_1\;\textsf{else}\;e_2}
   &\;\reduces\; \ehole{e_1} \\
\ehole{v_1 == v_2} &\;\reduces\; \ehole{b} \quad (b = (v_1 = v_2)) &
\ehole{\textsf{if}\;\textsf{false}\;\textsf{then}\;e_1\;\textsf{else}\;e_2}
   &\;\reduces\; \ehole{e_2}
\end{aligned}
\]
Here the notation is substitution $\subst{e}{x}{v}$ in the \textsf{let}
rather than map update.

\subsection{The running example, in context}

Take $E(t) = (\,\textsf{let}\;x = 1\;\textsf{in}\;x+1\,)\,;\,\eend$ after the
leading $\bbegin$ has been exposed; concretely the thread starts at
$E(t) = \bbegin \,;\, (\textsf{let}\;x = 1\;\textsf{in}\;x+1)\,;\,\eend$, which is
$\ehole{\bbegin}$ with
\[
\ectx \;=\; [\cdot]\,;\,(\textsf{let}\;x = 1\;\textsf{in}\;x+1)\,;\,\eend.
\]
Applying $\textsc{Begin}$:
\[
\hat{E}' = E'(t) = \ehole{\unit}
            = \unit \,;\, (\textsf{let}\;x = 1\;\textsf{in}\;x+1)\,;\,\eend,
\]
and the administrative rule $\ehole{\unit\,;\,e}\reduces \ehole{e}$ then drops the leading
$\unit$, leaving $(\textsf{let}\;x = 1\;\textsf{in}\;x+1)\,;\,\eend$ as both the
live expression and the saved resume point $\hat{E}$. The subsequent steps ::
$\textsf{let}$, then $+$, then committing at $\eend$, all proceed by plugging
into successively trailing \texttt{rest} contexts. Should a $\retry$ be reached anywhere in
the body, $\textsc{Retry}$ restores $E'(t)=\hat{E}
= (\textsf{let}\;x = 1\;\textsf{in}\;x+1)\,;\,\eend$, re-running the transaction
from just after $\bbegin$ without re-opening it.

\section{A declarative semantics for internal consistency}
\label{sec:declarative}

In a single-thread, mutual-exclusion, retry, $\forall$ program $P$: either $P$ never executes \texttt{retry}, or it executes \texttt{retry} infinitely often. I do not have a concrete proof of it, but the state the thread restarts from is $\sigma(t), \hat{E(t)}$, both fixed at \texttt{begin} and never mutated, and nothing else can change $\sigma$. Therefore, every attempt starts from an identical configuration; determinism makes every attempt an identical replay. With the constraint for our phase 1, I do not think we can have a ``retry exactly once'' mechanism. Also, the number of threads $n$ is irrelevant because the transaction holds the lock across its retries. For a ``retry exactly once'' mechanism, we at least need two threads in a transaction area. Let's call one a \texttt{waker}. For a waker to exist, some other thread must change $l$ between the re-attempts while $A$ is in mid-transaction. We can also consider one more thing: the \texttt{retry} releases the lock, which I have tried to capture in the \textsc{Retry-Alt}. It sets $\lambda(t) = \bot$, aborts and blocks, letting another transaction commit a change, and then re-acquires the lock and re-runs.

Following Cerone and Gotsman, we record one transactional run as a
\emph{history}: a set of events $e = (\mathit{id}, o)$ totally ordered by program
order $\po$, where each operation $o$ is a read $\rdev{l}{v}$, a write
$\wrev{l}{v}$, or a retry marker $\retryev$. We write $\HEvent_l$ for the read
and write events on location $l$, $\po^{-1}(e) = \{f \mid f \pob e\}$ for the
events preceding $e$, and $R(E)$ for the retry events of the run. Their internal
consistency axiom requires every read to return the value of the
program-order-latest operation on the same location that precedes it:
\[
\forall e.\ \forall l,v.\quad
\bigl(e = (\_,\rdev{l}{v}) \,\wedge\, \po^{-1}(e)\cap\HEvent_l \neq \emptyset\bigr)
\implies
\maxpo\bigl(\po^{-1}(e)\cap\HEvent_l\bigr) = (\_,\anyop{l}{v}),
\tag{\textsc{Int}}
\]
where $\anyop{l}{v}$ abbreviates ``a read \emph{or} a write of $v$ on $l$''.

Because \textsc{Retry-Alt} sends the thread back to $\bbegin$ --- discarding the
attempt's writes, releasing the lock, and re-reading from a \emph{fresh
snapshot} --- every read must be redone in the new attempt, and a read may no
longer depend on anything observed before the retry. We capture this by keeping,
for a read $e$, only the preceding operations on $l$ that lie in its
\emph{current attempt}: those not separated from $e$ by a retry event,
and  replacing the predecessor set of \textsc{Int} with $\att_l(e)$:
all events f that (1) precede e in program order, (2) touch l, and (3) have no retry between them and e.
